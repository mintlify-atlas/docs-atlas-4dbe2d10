title: 'Credential Formats Configuration'

The EUDI Verifier Endpoint supports multiple credential formats including SD-JWT VC and MSO MDoc. This page explains how to configure support for these formats and their validation settings.

## Supported Formats

The verifier can be configured to support:
- **SD-JWT VC** - Selective Disclosure JWT Verifiable Credentials
- **MSO MDoc** - Mobile Security Object Documents (ISO/IEC 18013-5)

## SD-JWT VC Configuration

### Enable SD-JWT VC Support

<ParamField path="VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_ENABLED" type="boolean" default="true">
  Enables support for SD-JWT VC credentials.
</ParamField>

### SD-JWT Signature Algorithms

<ParamField path="VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_SDJWTALGORITHMS" type="string" default="ES256">
  Comma separated list of signature algorithms the Issuer Signed JWT of an SD-JWT VC can be signed with.
  
  **Possible values:** Any `Algorithm Name` of an IANA registered asymmetric signature algorithm:
  https://www.iana.org/assignments/jose/jose.xhtml#web-signature-encryption-algorithms
  
  Common values:
  - `ES256` - ECDSA using P-256 and SHA-256
  - `ES384` - ECDSA using P-384 and SHA-384
  - `ES512` - ECDSA using P-521 and SHA-512
  - `RS256` - RSASSA-PKCS1-v1_5 using SHA-256
  - `PS256` - RSASSA-PSS using SHA-256
  - `EdDSA` - Edwards-curve Digital Signature Algorithm
</ParamField>

### Key Binding JWT Algorithms

<ParamField path="VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_KBJWTALGORITHMS" type="string" default="ES256">
  Comma separated list of signature algorithms the Key Binding JWT of an SD-JWT VC can be signed with.
  
  **Possible values:** Same as SD-JWT algorithms above.
</ParamField>

### Status Check Validation

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_STATUSCHECK_ENABLED" type="boolean" default="true">
  Enables status check validation for SD-JWT VC attestations shared.
  
  When enabled, the verifier will check credential status (e.g., revocation) using the status mechanisms defined in the credential.
</ParamField>

### SD-JWT VC Configuration Example

```bash
# Enable SD-JWT VC support
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_ENABLED=true

# Accept multiple signature algorithms
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_SDJWTALGORITHMS=ES256,ES384,EdDSA
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_KBJWTALGORITHMS=ES256,ES384,EdDSA

# Enable status checking
VERIFIER_VALIDATION_SDJWTVC_STATUSCHECK_ENABLED=true
```

## MSO MDoc Configuration

### Enable MSO MDoc Support

<ParamField path="VERIFIER_CLIENTMETADATA_VPFORMATS_MSOMDOC_ENABLED" type="boolean" default="true">
  Enable support for MSO MDoc credentials (ISO/IEC 18013-5).
</ParamField>

### MSO MDoc Configuration Example

```bash
# Enable MSO MDoc support
VERIFIER_CLIENTMETADATA_VPFORMATS_MSOMDOC_ENABLED=true
```

<Info>
  MSO MDoc credentials are validated according to ISO/IEC 18013-5 standards, including ValidityInfo expiration checks and IssuerSigned item digest verification.
</Info>

## SD-JWT VC Type Metadata Configuration

Type Metadata provides schema information and validation rules for SD-JWT VC credentials.

### Type Metadata Policy

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY" type="string" default="not_used">
  Choose SD-JWT VC Type Metadata policy.
  
  **Accepted values:**
  - `not_used` - Type Metadata is not used (default)
  - `optional` - Type Metadata is used if available
  - `always_required` - Type Metadata is required for all SD-JWT VCs
  - `required_for` - Type Metadata is required for specific VCTs
  
  Information about policies: https://github.com/eu-digital-identity-wallet/eudi-lib-jvm-sdjwt-kt?tab=readme-ov-file#type-metadata-resolution
</ParamField>

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY_REQUIREDFOR" type="string">
  Comma separated list of VCTs for which Type Metadata are required. 
  
  **Required when** `VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY` is set to `required_for`.
  
  Example: `urn:eudi:pid:1,urn:eudi:mdl:1`
</ParamField>

### Type Metadata Resolution

<Info>
  The following configuration is required when `VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY` is not set to `not_used`.
</Info>

Configure specific VCTs and their Type Metadata URLs using indexed environment variables:

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_0_VCT" type="string">
  VCT for which Type Metadata resolution is enabled.
  
  Example: `urn:eudi:pid:1`
  
  Use incrementing indices (0, 1, 2, ...) to configure multiple VCTs.
</ParamField>

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_0_URL" type="string">
  URL from which Type Metadata can be fetched for the corresponding VCT.
  
  Example: `http://localhost:8080/type-metadata/urn:eudi:pid:1`
</ParamField>

### Type Metadata Caching

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_CACHE_TTL" type="duration" default="PT1H">
  Cache TTL for resolved Type Metadata.
  
  Provide a value using Java Duration syntax.
  
  Examples:
  - `PT1H` - 1 hour (default)
  - `PT30M` - 30 minutes
  - `PT2H` - 2 hours
</ParamField>

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_CACHE_MAXENTRIES" type="integer" default="10">
  Cache maximum entries for resolved Type Metadata.
</ParamField>

### Type Metadata Integrity Validation

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_INTEGRITY_ENABLED" type="boolean" default="false">
  Enables sub-resource integrity validation for SD-JWT VC Type Metadata and JSON schemas.
  
  When enabled, the verifier will validate that fetched Type Metadata matches expected integrity hashes.
</ParamField>

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_INTEGRITY_ALLOWEDALGORITHMS" type="string" default="sha256,sha384,sha512">
  Comma-separated list of allowed sub-resource integrity hash algorithms.
  
  **Allowed values:** `sha256`, `sha384`, `sha512`
</ParamField>

### JSON Schema Validation

<ParamField path="VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_JSONSCHEMA_VALIDATION_ENABLED" type="boolean" default="true">
  Whether JSON Schema validation should be enabled or not.
  
  When enabled and Type Metadata includes a JSON Schema, the verifier will validate credential claims against the schema.
</ParamField>

## Configuration Examples

### Both Formats Enabled (Default)

```bash
# Enable both credential formats
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_ENABLED=true
VERIFIER_CLIENTMETADATA_VPFORMATS_MSOMDOC_ENABLED=true

# SD-JWT VC algorithms
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_SDJWTALGORITHMS=ES256
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_KBJWTALGORITHMS=ES256

# Enable status checking
VERIFIER_VALIDATION_SDJWTVC_STATUSCHECK_ENABLED=true

# No Type Metadata
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY=not_used
```

### SD-JWT VC Only with Type Metadata

```bash
# Enable only SD-JWT VC
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_ENABLED=true
VERIFIER_CLIENTMETADATA_VPFORMATS_MSOMDOC_ENABLED=false

# Accept multiple algorithms
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_SDJWTALGORITHMS=ES256,ES384,ES512,EdDSA
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_KBJWTALGORITHMS=ES256,ES384,ES512,EdDSA

# Type Metadata required for specific VCTs
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY=required_for
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY_REQUIREDFOR=urn:eudi:pid:1

# Configure Type Metadata URLs
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_0_VCT=urn:eudi:pid:1
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_0_URL=https://metadata.example.com/pid

# Type Metadata caching
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_CACHE_TTL=PT2H
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_CACHE_MAXENTRIES=20

# Enable integrity validation
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_INTEGRITY_ENABLED=true
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_INTEGRITY_ALLOWEDALGORITHMS=sha256,sha384

# Enable JSON Schema validation
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_JSONSCHEMA_VALIDATION_ENABLED=true
```

### MSO MDoc Only

```bash
# Enable only MSO MDoc
VERIFIER_CLIENTMETADATA_VPFORMATS_SDJWTVC_ENABLED=false
VERIFIER_CLIENTMETADATA_VPFORMATS_MSOMDOC_ENABLED=true
```

### Multiple Type Metadata Sources

```bash
# Type Metadata always required
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_POLICY=always_required

# Configure multiple VCT Type Metadata sources
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_0_VCT=urn:eudi:pid:1
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_0_URL=https://metadata.example.com/pid

VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_1_VCT=urn:eudi:mdl:1
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_1_URL=https://metadata.example.com/mdl

VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_2_VCT=urn:eudi:qeaa:1
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_VCTS_2_URL=https://metadata.example.com/qeaa

# Caching configuration
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_CACHE_TTL=PT1H
VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_CACHE_MAXENTRIES=50
```

## Algorithm Selection

### Recommended Algorithms for SD-JWT VC

| Algorithm | Security | Performance | Use Case |
|-----------|----------|-------------|----------|
| `ES256` | High | Excellent | General purpose (recommended) |
| `ES384` | Higher | Good | High security requirements |
| `ES512` | Highest | Good | Maximum security |
| `EdDSA` | High | Excellent | Modern cryptography |
| `PS256` | High | Moderate | RSA-based systems |
| `RS256` | Moderate | Moderate | Legacy compatibility |

<Tip>
  `ES256` and `EdDSA` offer the best balance of security and performance for most use cases.
</Tip>

## Type Metadata Policy Guide

<AccordionGroup>
  <Accordion title="not_used - No Type Metadata">
    **When to use:**
    - Simple deployments without schema validation
    - Testing and development
    - When issuers don't provide Type Metadata
    
    **Pros:** Simple configuration, no external dependencies
    
    **Cons:** No schema validation, less structured validation
  </Accordion>
  
  <Accordion title="optional - Best Effort">
    **When to use:**
    - Mixed environments with varying issuer capabilities
    - Gradual rollout of Type Metadata support
    - When backwards compatibility is needed
    
    **Pros:** Flexible, supports both old and new credentials
    
    **Cons:** Inconsistent validation behavior
  </Accordion>
  
  <Accordion title="always_required - Strict Validation">
    **When to use:**
    - Production environments with schema requirements
    - High security deployments
    - When all issuers provide Type Metadata
    
    **Pros:** Consistent validation, strong guarantees
    
    **Cons:** Requires all issuers to provide Type Metadata
  </Accordion>
  
  <Accordion title="required_for - Selective Enforcement">
    **When to use:**
    - Specific credential types need schema validation
    - Mixed credential type environment
    - Gradual enforcement of Type Metadata
    
    **Pros:** Flexible enforcement per credential type
    
    **Cons:** More complex configuration
  </Accordion>
</AccordionGroup>

## Security Best Practices

<AccordionGroup>
  <Accordion title="Enable Status Checking">
    Always enable `VERIFIER_VALIDATION_SDJWTVC_STATUSCHECK_ENABLED` in production to detect revoked credentials.
  </Accordion>
  
  <Accordion title="Use Strong Algorithms">
    Prefer `ES256` or stronger. Avoid `RS256` unless required for compatibility.
  </Accordion>
  
  <Accordion title="Enable Type Metadata in Production">
    Use `optional` or `always_required` policies in production for better validation.
  </Accordion>
  
  <Accordion title="Enable Integrity Validation">
    Set `VERIFIER_VALIDATION_SDJWTVC_TYPEMETADATA_RESOLUTION_INTEGRITY_ENABLED=true` when using Type Metadata.
  </Accordion>
  
  <Accordion title="Configure Appropriate Cache TTL">
    Balance between freshness and performance. 1-2 hours is typically appropriate.
  </Accordion>
</AccordionGroup>
