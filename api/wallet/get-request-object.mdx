## Overview

This endpoint allows the wallet to retrieve the OpenID4VP authorization request object when using the `request_uri_method: get` approach. The request object is returned as a signed JWT according to the JWT Secured Authorization Request specification.

## Endpoint

```
GET /wallet/request.jwt/{requestId}
```

## Path Parameters

<ParamField path="requestId" type="string" required>
  The unique identifier of the authorization request. This value is provided in the `request_uri` from the Initialize Transaction endpoint response.
</ParamField>

## Response

The response is a signed JWT (JSON Web Token) containing the authorization request payload.

**Content-Type:** `application/oauth-authz-req+jwt`

### Response Fields

The JWT payload contains the OpenID4VP authorization request with fields such as:

<ResponseField name="client_id" type="string">
  The client identifier of the verifier
</ResponseField>

<ResponseField name="response_type" type="string">
  Always `vp_token` for OpenID4VP flows
</ResponseField>

<ResponseField name="response_mode" type="string">
  Either `direct_post` or `direct_post.jwt`
</ResponseField>

<ResponseField name="state" type="string">
  A unique state value that must be included in the wallet's response
</ResponseField>

<ResponseField name="nonce" type="string">
  A unique nonce value to prevent replay attacks
</ResponseField>

<ResponseField name="presentation_definition" type="object">
  The DCQL query defining which credentials are requested
</ResponseField>

<ResponseField name="response_uri" type="string">
  The URL where the wallet should POST its response
</ResponseField>

## Error Responses

<ResponseField name="404 Not Found" type="error">
  The request object with the specified `requestId` was not found or has expired
</ResponseField>

<ResponseField name="400 Bad Request" type="error">
  The request is malformed or invalid
</ResponseField>

## Usage Flow

1. Verifier initializes a transaction and receives a `request_uri`
2. Verifier presents the `request_uri` to the wallet (via QR code or deep link)
3. Wallet extracts the `requestId` from the `request_uri`
4. Wallet calls this GET endpoint to retrieve the authorization request JWT
5. Wallet validates and processes the JWT to determine which credentials to present
